# 仕様ファースト開発ガイド

> 新機能を実装する前に、必ず仕様書を作成する運用ルールです。

---

## なぜ仕様ファーストか？

Boris Cherny（Claude Code 開発者）の推奨に基づいています：

> 「新機能を作るときは .md ファイルに仕様を書いてから Claude に実装させる。
> この仕様書が最終的にそのサブプロジェクトの CLAUDE.md になる。」

### メリット
1. **思考の整理** - 書き出すことで要件が明確になる
2. **AI への明確な指示** - 曖昧さを減らし、手戻りを防ぐ
3. **ドキュメントの自動生成** - 仕様書がそのままドキュメントになる
4. **変更履歴** - Git で仕様の変遷を追跡できる

---

## 運用フロー

```
1. 仕様書を作成
   └── docs/specs/feature-name.md

2. Claude に渡して実装
   └── "docs/specs/feature-name.md を読んで実装して"

3. 実装中に仕様を更新
   └── 新しい発見や変更点を追記

4. 完了後
   └── ステータスを "Completed" に変更
   └── 仕様書は参照ドキュメントとして残す
```

---

## テンプレートの使い方

1. `_template.md` をコピー
2. `feature-name.md` にリネーム
3. 各セクションを埋める（すべて埋める必要はない）

```bash
cp docs/specs/_template.md docs/specs/my-new-feature.md
```

---

## ファイル命名規則

```
docs/specs/
├── _template.md              # テンプレート（コピー元）
├── gallery-animation.md      # 例：ギャラリーアニメーション機能
├── timeline-filter.md        # 例：タイムラインフィルター機能
└── README.md                 # このファイル
```

- ケバブケース（kebab-case）を使用
- 機能を端的に表す名前

---

## 仕様書のステータス

| ステータス | 意味 |
|-----------|------|
| **Draft** | 下書き中、まだ実装開始前 |
| **In Progress** | 実装中 |
| **Completed** | 実装完了 |
| **Cancelled** | 実装しないことに決定 |

---

## Tips

- **完璧を求めない** - 最初は概要だけでもOK
- **実装中に更新** - 仕様は生きたドキュメント
- **コードリンクを追加** - 実装後、関連ファイルへのリンクを追記すると便利
